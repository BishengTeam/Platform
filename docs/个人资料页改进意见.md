## 三、发现的问题

### 🔴 问题 1（高）：`id_card` 被前端发送到 PUT 接口，但后端 schema 不接收

**现象**：

`edit-profile.tsx` 的 `handleSave` 函数在提交时将 `id_card` 作为参数传给了 `updateUserProfile`：

```typescript
// src/pages/mine/edit-profile.tsx — handleSave()
await updateUserProfile({
  email,
  gender,
  education,
  id_card: idCard,  // ← 发送了 id_card
  ...(isStudent ? { school, major } : { organization }),
})
```

`userService.ts` 的类型签名中也明确声明了 `id_card?: string`：

```typescript
// src/services/userService.ts:291
export async function updateUserProfile(data: {
  email?: string
  gender?: string
  education?: string
  school?: string
  major?: string
  organization?: string
  id_card?: string        // ← 声明了，但后端不接受
}) {
```

**而后端 `UserProfileUpdate` schema 不含 `id_card`**：

```python
# Backend/app/schemas/user.py:98-105
class UserProfileUpdate(BaseModel):
    phone: str | None = Field(None, min_length=11, max_length=11, description="手机号")
    email: str | None = Field(None, max_length=128, description="邮箱")
    gender: str | None = Field(None, max_length=8, description="性别")
    education: str | None = Field(None, max_length=32, description="学历")
    school: str | None = Field(None, max_length=128, description="学校")
    major: str | None = Field(None, max_length=128, description="专业")
    organization: str | None = Field(None, max_length=256, description="单位")
    # ❌ 没有 id_card
```

**影响**：

后端 `update_profile` 方法通过 `data.model_dump(exclude_unset=True)` 提取字段，Pydantic 只会提取 schema 中定义的名字。`id_card` 不在 `UserProfileUpdate` 中 → **被静默丢弃**。前端以为修改了身份证号，实际请求中该字段被忽略。

**触发条件**：

`edit-profile.tsx` 中 `isIdCardDisabled` 的判断逻辑为：

```typescript
const isIdCardDisabled = identityStatus === 'verified' || identityStatus === 'pending'
```

即只有 `verified` 和 `pending` 状态下身份证号输入框才被禁用。当 `identity_status` 为 `null`、`undefined` 或 `rejected` 时，用户可以编辑身份证号并提交，**但提交的值不会生效**。

**修复方向**（二选一）：

- **方案 A（推荐）**：业务上不允许通过此接口修改身份证号 → 前端删除 `updateUserProfile` 中的 `id_card` 参数；`edit-profile.tsx` 中身份证号始终设为 `disabled`。
- **方案 B**：业务上允许修改 → 后端 `UserProfileUpdate` 新增 `id_card` 字段，并在 `user_service.update_profile` 中处理该字段的更新逻辑。

### 🟡 问题 2（中）：`id_card` 返回值是脱敏数据，现有文档描述有误

`个人资料页修改计划.md` 中对 `GET /api/user/profile` 的字段描述写着：

```
id_card: string — '110101199001011234'  (明文)
```

**实际情况**：后端 `user_service.get_profile()` 对 `id_card` 做了脱敏处理：

```python
# Backend/app/services/user.py:142-145
id_card = None
if identity and identity.id_card_number:
    idn = identity.id_card_number
    id_card = idn[:4] + "**********" + idn[-4:] if len(idn) == 18 else "****"
```

返回的 `id_card` 是 `1101**********1234` 这样的脱敏值，**不是明文**。明文仅在 `id_card_raw` 中且仅当 `identity_status === 'verified'` 时才返回。

**影响**：`personal-info.tsx` 展示页直接从 `profile.id_card` 取值展示没有问题（展示脱敏值是正确的）。`edit-profile.tsx` 中：

```typescript
setIdCard(profile.id_card_raw || profile.id_card || '')
```

采用了 `id_card_raw` 优先的 fallback 策略，逻辑基本正确。但当用户未实名时 `id_card_raw` 为 `null`，回退显示脱敏值，此时如果用户未被禁止编辑（identityStatus 不是 verified/pending），会把脱敏值传回 PUT → 而且即使传了也不生效（问题 1）。

**建议**：修正 `个人资料页修改计划.md` 第 22 行的文档错误。

### 🟡 问题 3（中）：前端缺少 Profile 响应的强类型定义

`getUserProfile()` 和 `updateUserProfile()` 均使用 `get<any>` / `put<any>`，没有定义与后端 `UserProfileDetail` 对应的 TypeScript 接口：

```typescript
// userService.ts
const res = await get<any>('/api/user/profile')   // ← any
return res.data
```

`types/profile.ts` 文件只定义了 `OrderItem`、`ProfileFunction`、`ProfileMenuItem`，缺少 `UserProfileDetail` 接口。

**影响**：
- 字段名拼写错误编译期无法发现
- 后端新增/修改字段前端无感知
- 不利于后续报名页等需要消费 `pinyin`、`age`、`phone_raw` 等扩展字段的场景

**建议**：在 `src/types/profile.ts` 中新增：

```typescript
/** 对应后端 GET /api/user/profile 返回值 */
export interface UserProfileDetail {
  id: number
  openid: string
  phone: string | null
  email: string | null
  real_name: string | null
  id_card: string | null
  user_type: string | null
  gender: string | null
  education: string | null
  school: string | null
  major: string | null
  organization: string | null
  identity_status: string | null
  created_at: string
  phone_raw: string | null
  id_card_raw: string | null
  pinyin: string | null
  first_name: string | null
  last_name: string | null
  age: number | null
  country: string
  language: string
}

/** 对应后端 PUT /api/user/profile 请求体 */
export interface UserProfileUpdatePayload {
  phone?: string
  email?: string
  gender?: string
  education?: string
  school?: string
  major?: string
  organization?: string
}
```

然后将 `getUserProfile` 和 `updateUserProfile` 的返回值改为强类型。

### 🟡 问题 4（低）：`edit-profile.tsx` 存在无效字段"昵称"

`edit-profile.tsx` 中声明了 `nickname` 状态并在表单中渲染了"昵称"输入框：

```typescript
// 第 17 行
const [nickname, setNickname] = useState('')

// 第 39-40 行：从 real_name 赋值，但后端无 nickname 概念
setNickname(profile.real_name || '')

// 第 97 行
<FormInput label='昵称' placeholder='' value={nickname} onChange={setNickname} disabled={isReadonly} />
```

但：
- 后端 `UserProfileDetail` 和 `UserProfileUpdate` 中**均没有** `nickname` 字段
- `handleSave` 中**未提交** `nickname`

**影响**：用户可以看到并编辑"昵称"，但保存后不会持久化，属于 UI 残留。

**建议**：移除 `nickname` 相关状态、输入框和赋值逻辑。
