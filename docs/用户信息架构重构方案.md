> 日期：2026-06-10
> 状态：待评审

## 一、当前结构（重构前）

```
user                             user_identity
├── id                           ├── user_id (FK→user.id, UNIQUE)
├── openid                       ├── user_type (student/enterprise)
├── phone                        ├── real_name
├── is_active                    ├── id_card_number
└── created_at/updated_at        ├── id_card_front_oss
                                 ├── id_card_back_oss
                                 ├── student_card_oss
                                 ├── email
                                 ├── education
                                 ├── gender
                                 ├── school
                                 ├── major
                                 ├── organization
                                 ├── status (unsubmitted/pending/verified/rejected)
                                 ├── edit_count
                                 └── verified_at
```

**问题**：
- 一张 `user_identity` 混了实名、教育、企业、邮箱等多种语义
- 没有审核分级——修改任何字段都会立即生效，`status` 只用于实名认证场景
- 企业/学生差异化字段不强约束

## 三、审核机制

```
用户修改 Level-1 字段 (PUT /api/user/profile)
  → 直接写 user_profile 表
  → 旧值被覆盖

用户修改 Level-2 字段 (POST /api/user/identity, PUT /api/user/student ...)
  → 写入对应子表，status = "pending"
  → 管理端展示「待审核」标记
  → 管理员审核通过 → status = "verified" + verified_at = now
  → 管理员驳回 → status = "rejected"，前端展示旧值不变

同时存在 "pending" 和 "verified" 记录时的读取规则：
  → 对外展示：用 verified 时的值（即审核通过后的值）
  → 管理端审核页：展示 pending 时的值
  → 实现方式：pending 写入 pending 字段（或独立 pending 表）
    管理员审核通过后拷贝到主字段并清空 pending

简化方案（建议）：
  → 子表只存一份数据
  → status 管理审核状态
  → "pending" 期间：读取侧返回旧值（不展示未审核数据）
  → 需要维护一份「上次审核通过时的快照」——可以在审核通过时
    将字段值备份到 verified_* 字段，或使用独立的 snapshot 表
```

**关键设计问题**：pending 期间用户看到什么？

| 方案 | 描述 | 复杂度 |
|------|------|--------|
| A: 覆盖写 + 快照表 | pending 直接覆盖主字段，审核通过无操作，驳回时从快照恢复 | 中 |
| B: 双写 | 主字段始终存 verified 值，pending 字段存待审核值，审核通过后主字段←pending | 低 |
| C: 不改动 + 乐观 | 用户提交后立即生效（当前行为），管理员可手动驳回，驳回后主字段不恢复 | 最低 |

## 四、API 变更

### 4.1 用户端

| 方法 | 路径 | 当前 | 重构后 |
|------|------|------|--------|
| GET | `/api/user/profile` | UserProfileDetail | 返回 user + user_profile + user_realname + user_student/enterprise 聚合 |
| PUT | `/api/user/profile` | 改 phone, email, gender, school... | **仅改 Level-1 字段**：nickname, email, phone |
| POST | `/api/user/identity` | 提交实名 | 提交 Level-2 实名信息（触发审核） |
| PUT | `/api/user/identity` | 无 | 🆕 修改实名信息（重新提交审核） |
| POST | `/api/user/student` | 无 | 🆕 提交学生信息（触发审核） |
| PUT | `/api/user/student` | 无 | 🆕 修改学生信息（重新提交审核） |
| POST | `/api/user/enterprise` | 无 | 🆕 提交企业信息（触发审核） |
| PUT | `/api/user/enterprise` | 无 | 🆕 修改企业信息（重新提交审核） |

### 4.2 管理端

| 方法 | 路径 | 当前 | 重构后 |
|------|------|------|--------|
| GET | `/admin/users/{id}/profile` | 聚合 | 同用户端聚合，但手机/身份证不脱敏 |
| PUT | `/admin/users/{id}/profile` | 改所有字段 | 🗑 删除（管理员不应直接修改用户敏感信息） |
| PUT | `/admin/users/{id}/identity/review` | 审核实名 | 审核实名信息 |
| PUT | `/admin/users/{id}/student/review` | 无 | 🆕 审核学生信息 |
| PUT | `/admin/users/{id}/enterprise/review` | 无 | 🆕 审核企业信息 |

## 五、数据库迁移动作

```
1. user 表
   - DROP COLUMN phone            → 迁到 user_profile

2. 重命名 user_identity → user_profile
   - KEEP: user_id, email
   - ADD:  nickname (VARCHAR 64)
   - DROP: user_type, real_name, id_card_number, id_card_front_oss,
           id_card_back_oss, student_card_oss, education, gender,
           school, major, organization, status, edit_count, verified_at

3. 新建 user_realname
   - user_id, user_type, real_name, id_card_number,
     id_card_front_oss, id_card_back_oss,
     gender, age, census_register (全自动计算),
     status, edit_count, verified_at, timestamps

4. 新建 user_student
   - user_id, education, school, major, student_card_oss,
     status, edit_count, verified_at, timestamps

5. 新建 user_enterprise
   - user_id, organization,
     status, edit_count, verified_at, timestamps

6. 存量数据迁移脚本
   - user_identity.email → user_profile.email
   - user.phone → user_profile.phone
   - user_identity.real_name, id_card_number, ... → user_realname
   - user_identity.education, school, major, student_card_oss → user_student
   - user_identity.organization → user_enterprise
   - 初始 status = "verified"（存量数据视为已审核）
```

## 七、待决问题（已确认）

| # | 问题 | 决定 |
|---|------|------|
| 1 | pending 期间读取策略 | 用户立即看到新值；驳回时从快照恢复旧值 |
| 2 | user_realname 创建时机 | 首次提交实名时创建（不在登录时创建） |
| 3 | user_student / user_enterprise 关系 | 互斥，只能存一 |
| 4 | 存量数据迁移策略 | 一次性迁移 |
| 5 | edit_count 策略 | 全局共享（放 user 表），上限 5 次，30 天刷新 |

### edit_count 实现

```
user 表新增：
  level2_edit_count  INT  DEFAULT 0
  level2_edit_reset  TIMESTAMPTZ

任意 Level-2 接口（POST/PUT identity/student/enterprise）：
  1. 检查 level2_edit_reset 是否超过 30 天 → 是则归零
  2. 检查 level2_edit_count >= 5 → 拒绝
  3. 写入对应子表 + level2_edit_count += 1
```
