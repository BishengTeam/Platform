# 实名认证状态设计说明

> 日期：2026-06-09

---

## 1. 设计原则

`verified` 是用户敏感功能的**单一开关**。只有审核通过的用户才能使用涉及身份证号、手机号明文等敏感信息的功能。

---

## 2. 三种状态

| 状态 | 含义 | 触发条件 | 可调用敏感功能 |
|------|------|---------|:---:|
| `pending` | 待审核 | 首次提交 / 核验服务不可用 / 未配置核验 | ❌ |
| `verified` | 已通过 | 核验 API 明确返回匹配 / 管理员手动审核 | ✅ |
| `rejected` | 已拒绝 | 核验 API 明确返回不匹配 | ❌ |

---

## 3. 个人信息接口的分层返回策略

`GET /api/user/profile` 根据 `identity_status` 返回不同粒度的数据：

### 3.1 始终返回（脱敏或基本信息）

| 字段 | 说明 |
|------|------|
| `real_name` | 真实姓名（明文，用于页面展示） |
| `phone` | 手机号脱敏（`138****0001`） |
| `id_card` | 身份证号脱敏（`1101**********1234`） |
| `email` / `gender` / `education` 等 | 基本资料（明文） |
| `identity_status` | 当前审核状态 |

### 3.2 仅 `verified` 时返回（明文，用于自动预填）

| 字段 | 说明 |
|------|------|
| `phone_raw` | 明文手机号 |
| `id_card_raw` | 明文身份证号 |
| `age` | 从身份证号推算的年龄 |
| `pinyin` / `first_name` / `last_name` | 姓名拼音（用于报名表单） |

---

## 4. 安全边界

```
                    脱敏区                    |          明文区
                                             |
  GET /api/user/profile                      |  GET /api/user/profile
  GET /api/user/identity                     |  (仅 identity_status == "verified")
                                             |
  phone: "138****0001"                       |  phone_raw: "13800000001"
  id_card: "1101**********1234"              |  id_card_raw: "110101199001011234"
  id_card_number: "1101**********1234"       |
                                             |
  任何拿到 token 的用户都可以看到             |  仅审核通过的用户可以看到
  用于页面展示、确认身份                      |  用于认证报名自动预填
```

---

## 5. 前端使用规范

| 场景 | 数据来源 | 说明 |
|------|---------|------|
| 页面展示用户信息 | `phone` / `id_card`（脱敏） | 安全，可直接渲染 |
| 认证报名自动预填 | `phone_raw` / `id_card_raw` | 仅 `verified` 时可用 |
| 提交实名认证 | 用户手动输入 | **禁止回填脱敏数据** |
| 手机号重新获取 | 微信 `getPhoneNumber` 授权 | 与 profile 无关 |

### ❌ 错误用法

```
GET /api/user/identity → id_card_number: "1101**********1234"
                                ↓
                        前端预填到表单
                                ↓
                        POST /api/user/identity  →  422  (含 * 号)
```

### ✅ 正确用法

```
GET /api/user/profile → identity_status == "verified"
                                ↓
                        id_card_raw: "110101199001011234"
                                ↓
                        前端预填到表单 → 用户确认 → 提交成功
```

---

## 6. `rejected` 与 `pending` 的处理

两者均不返回明文敏感字段。区别在于业务语义：

- `pending`：用户可等待审核，无需操作
- `rejected`：信息不匹配，用户应在 3 次 `edit_count` 限制内重新提交

---

## 7. 后续优化方向

| 方向 | 说明 |
|------|------|
| `pending` 自动审核 | 对接阿里云/腾讯云核验 API（已预留 `app/integrations/identity_verify.py`） |
| `rejected` 引导 | 返回被拒原因，引导用户修正后重新提交 |
| 敏感功能清单 | 竞赛报名、认证报名等需要 `verified` 的功能统一加 `require_identity` 中间件 |
