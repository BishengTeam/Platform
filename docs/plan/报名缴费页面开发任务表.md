# 报名缴费页面开发任务表（两人并行）

> 基于 [小程序前端页面设计文档 V1](./小程序前端页面设计V1.md) 第 3 章节
> 涉及页面：报名专区 → 报名表单 → 报名确认 → 支付结果

---

## 页面范围

| 页面 | 路径 | 核心功能 |
|------|------|----------|
| 报名专区 | `pages/registration/index` | 搜索栏 + 分类Tab + 认证卡片列表 |
| 报名表单 | `pages/registration/form` | 表单校验 + 考试券条件显隐 + 费用明细 |
| 报名确认 | `pages/registration/confirm` | 订单汇总 + 微信支付调用 + 协议勾选 |
| 支付结果 | `pages/payment/result` | 成功/失败双状态展示 |

**依赖链**：`index → form → confirm → result`，页面间通过 URL 参数传递数据。

## 联调阶段

| # | 任务 | 负责人 |
|---|------|--------|
| L1 | 对接 `GET /api/certifications`（报名列表） | A |
| L2 | 对接 `POST /api/orders` + `POST /api/payment/prepay`（下单+支付） | B |
| L3 | 对接 `GET /api/orders/{id}`（订单状态查询） | B |
| L4 | 全流程走通：首页 → 报名专区 → 表单 → 确认 → 支付 → 结果 | 共同 |

## 并行策略

```
          P0-P2 (共同先做)
          /            \
    A1-A4               B1-B2
    (index页)           (confirm骨架+数据展示)
      |                    |
    A5-A8               B3-B4
    (form页)            (支付逻辑+协议)
      |                    |
      └──── 联调对接 ──────┘
              |
         L1-L4 (共同)
```

- **A 和 B 的第 1-2 轮完全独立**，无阻塞。
- A 的 form 页需要接收 index 页传递的 `cert_type`，这是页面内闭环，不影响 B。
- B 的 confirm 页需要接收 form 页传递的表单数据，**可先用 mock 数据自测**，等 A7 完成后再联调打通。
- B 的 result 页需要 confirm 页支付后的回调，同样可先用 mock 参数自测。

```js
// mock 表单数据（代替 A 的 form 页传递），confirm 页 onLoad 中使用
const mockFormData = {

  cert_name: 'H3CNE-RS+ 认证工程师',
  real_name: '张三',
  phone: '138****8888',
  id_card: '3301**********1234',
  identity_type: 'enterprise',
  price: 1200,
  coupon_count: 0,
  enterprise_name: '',

```

B 开发 confirm 和 result 页时，先 `onLoad` 中 hardcode 这份 mock 数据完成自测，联调时替换为真实的 `options` 参数解析即可。
